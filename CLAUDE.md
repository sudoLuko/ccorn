# CCorn

Native macOS menu-bar app that manages Claude Code sessions running in tmux. Swift and SwiftUI, macOS 13+. It is a process manager, not a chat interface: it starts, watches, and controls sessions, and you keep talking to Claude in your terminal or at claude.ai/code.

## Read first

- `docs/CCORN_SPEC.md` is the full build spec (architecture, design language, every screen and flow). Read the sections relevant to your task before writing code. This file is the quick orientation; the spec is the source of truth.
- `README.md` for the user-facing overview and the status-color table.
- `RELEASING.md` for how a signed, notarized build is cut.

## Build, test, run

The Xcode project is generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml`. `CCorn.xcodeproj` is git-ignored and must not be hand-edited; change `project.yml` and regenerate. After adding, removing, or moving any Swift file, run `xcodegen generate` before building.

```sh
brew install xcodegen tmux                    # one-time
xcodegen generate                             # regenerate the project from project.yml
xcodebuild -project CCorn.xcodeproj -scheme CCorn -configuration Debug build
xcodebuild test -project CCorn.xcodeproj -scheme CCorn -destination 'platform=macOS'
```

A full Xcode install is required (Command Line Tools alone cannot build the app bundle). The Debug config disables the hardened runtime to keep lldb attach and the debug seams working; Release enables it for notarization.

Debug builds expose a file-based command channel for scripted, headless verification (`CCORN_DEBUG_UI=cmd`, polls `/tmp/ccorn-debug-cmd`; see `Sources/UI/DebugCommandChannel.swift`). All debug seams are behind `#if DEBUG` and compiled out of Release.

`project.yml` defines two targets: `CCorn` (the app) and `CCornEngineTests` (a hostless logic-test bundle that compiles the engine's pure-logic sources directly, so tests run with no status item or window server). When a test needs an engine source, add it to the test target's `sources` list in `project.yml`, not just to the test file.

## Architecture overview

The engine (`SessionEngine`) owns a single tmux session named `ccorn`, with one window per Claude Code session. It starts, resumes, and terminates sessions, tracks live PIDs, and reconciles with existing tmux windows on launch. `AppModel` is the UI-facing coordinator on top: it owns the 3s state poll, the FSEvents-driven discovery refresh, the row models every screen renders, and the full action surface (new, import, kill, restart, archive, rename). The engine and its store are `@MainActor` isolated and observable, so SwiftUI reads and the poll timer's writes never race; tmux/process/disk work runs in detached tasks on value snapshots and hops results back to the main actor.

Key patterns a contributor needs:

- **Durable session identity.** The Claude session UUID (the JSONL filename, equal to the `sessionId` field) is the identity key, never a hardcoded id. Each tmux window is tagged with `@ccorn_id <uuid>`; target programmatic commands (`send-keys`, `capture-pane`, `kill-window`, `rename-window`) by the stable window id (`@N`) or that tag, resolving any name to an id first. The tmux window name is a display/attach label only. PIDs, state, and pane hashes are runtime-only and re-derived on every launch; only the archived flag, UUID, last-known path, and title are persisted (JSON under `~/Library/Application Support/CCorn/`).
- **Discovery.** Primary source is `~/.claude/projects/`, not a project-local `.claude/` folder (often absent). The directory name is a lossy encoded path; resolve the real path from the `cwd` field on the first transcript line that carries one (line 1 is a metadata record without `cwd`). Transcripts are created lazily, after the first turn. Ignore the sibling `memory/` directory. Match watch dirs against `cwd` by resolved absolute path (canonicalize symlinks first).
- **Process identification.** Match the `claude` process by argv (argv[0] basename `claude`, or args containing `--rc`) or exec-path basename, never by `proc_name`/`p_comm`/`ps -o ucomm` (a native install is version-named, e.g. `2.1.169`). Match only among children of the shell CCorn spawned; never a global `pgrep claude`.
- **Status-state model.** `SessionState` is the lifecycle enum; `StatusPresentation` is the one mark a row shows. Every row shows exactly one mark: a colored dot for routine states, or the single `exclamationmark.triangle.fill` (amber recoverable, red terminal) for the broken trio (sign-in, no-remote, crashed). Never both, never any other status glyph. Status marks are the only color in the app.
- **Template menu-bar image, aggregate state in the popover.** `CornIcon.menuBarImage` is a single-color imageset always rendered as a template, so macOS tints it to follow the menu-bar appearance. It never carries state color (a template image cannot). The aggregate "most urgent" mark across all managed sessions is computed by `AppModel.aggregatePresentation` and shown in the popover header, not on the menu-bar icon.
- **Motion gating.** Row status-mark animation is gated on a `rowMotionEnabled` SwiftUI environment value driven by whether the hosting window is on screen (see `Theme.swift`, `MainWindowController`). When a window leaves the screen the repeating motion stops, and restarts on the next show. Honor this gate; do not start unconditional `repeatForever` animations.
- **Debug seams.** `Sources/Debug/` and `DebugCommandChannel`/`DebugStage`/`DebugLifecycle` are the scripted-verification and design-staging surface, all `#if DEBUG`. Seeding goes through `AppModel.debugSeed`, which stops the live poll first and never touches the real store, tmux session, or discovery.

## Execution environment (hard constraints)

- **PATH.** A GUI app launched from Finder/Xcode does not inherit the shell PATH, so `tmux`, `claude`, and `brew` (in `/opt/homebrew/bin` or `/usr/local/bin`) are not found. Run external commands through a login shell or resolved absolute paths. Use one resolution strategy for every `Process` invocation.
- **App Sandbox is OFF, on purpose.** CCorn spawns `tmux`/`claude`/`ps`/`lsof`/`osascript`, watches arbitrary directories with FSEvents, and sends AppleEvents to Terminal, none of which a sandboxed app may do. The entitlements file exists only for the hardened runtime's AppleEvents permission; it does not enable the sandbox. Do not add the App Sandbox capability.
- **Activation policy.** Default `.accessory` (no Dock icon, no Cmd+Tab). Switch to `.regular` when a regular window opens (main window, Settings, onboarding) and back to `.accessory` only when none remains. Do not set `LSUIElement`; it breaks the dynamic switch.
- **Remote control.** Sessions are remote by default: pass `--rc` and a title (`claude --rc "<title>"`, `claude --resume <uuid> --rc`). The New Session sheet can launch a **local** session instead (`SessionLaunchConfig.remoteControl == false`): no `--rc`, no remote/phone access, no per-session URL. The choice is persisted on the record (so a restart honors it) and consulted by `StatusPresentation.resolve` (so a local session is never flagged no-remote). `--rc` is positional, not a `claudeFlagTokens()` flag: the engine folds it into the base invocation via `SessionEngine.claudeBase`. Detect remote-control-active via the `Remote Control active` footer string or a `bridge-session` record in the session JSONL. "Open in Browser" opens https://claude.ai/code (the session list). Do not write a global remote-control key to `~/.claude.json`.

## tmux reference

- Check, then create only if absent: `tmux has-session -t ccorn` / `tmux new-session -d -s ccorn`.
- New window (capture the id): `tmux new-window -t ccorn -n <sanitized-name> -c <dir> -P -F "#{window_id}"`.
- Start / resume: `tmux send-keys -t <window-id> 'claude --rc "<title>"' Enter` / `'claude --resume <uuid> --rc' Enter`. Send the key as a separate `Enter` argument, never an embedded newline.
- Capture the visible frame (alt-screen TUI has no scrollback): `tmux capture-pane -t <window-id> -p`.
- Terminate routine: `kill-window`, then if the PID is still alive `SIGTERM`, wait 5s, then `SIGKILL`.

State detection still polls pane output every 3s; everything else (discovery, new sessions) is FSEvents-driven, not polled.

## Code style and conventions

- **Colors.** Use SwiftUI semantic colors (`Color.primary`, `Color.secondary`, `Color(.separatorColor)`, `Color(.windowBackgroundColor)`) for all main-window UI; never hardcode hex there. Hardcoded hex is confined to the status palette and the menu-bar popover (`PopoverPalette`), both appearance-paired via `Color(lightHex:darkHex:)`; the popover follows the app appearance (its dark face is the original `#09090B`), and only the template menu-bar icon stays independent. Status tokens are explicit hex, not semantic colors: some are appearance-paired via `Color(lightHex:darkHex:)` (green, blue, attention amber, stopped outline), the rest are identical in both appearances. Exact tokens are in `Theme.swift` and `docs/CCORN_SPEC.md` section 3, and are pinned by `ThemeTokenTests`/`StatusPresentationTests`.
- **Type and weight.** SF Pro Text; SF Mono for directory paths via `.monospaced()`. Two font weights only, regular (400) and medium (500), never bold or semibold. App identity is the OpenMoji corn glyph (`CornMark`), not a wordmark: it sits in the main-window title bar and the popover header, with no "CCorn" text alongside. Prefer named SwiftUI text styles (they respect Dynamic Type); `.system(size:)` appears only for sub-caption sizes that have no named style.
- **Borders and decoration.** 0.5px borders only, never 1px. No shadows, gradients, or decorative elements. Exception: hollow status-dot rings (stopped/unmanaged) are 1px, because a 0.5px ring is invisible at 7px.
- **SwiftUI.** `.listStyle(.sidebar)` for the sidebar; do not hardcode row heights. Use the `Settings` scene for preferences (no custom window, no `NavigationStack` inside it). `NSMenu` for context menus, not custom SwiftUI menus. Native `NSOpenPanel` for folder picking.
- Match the surrounding code's naming and comment density. Explain non-obvious platform behavior where it is decided, the way the existing files do.

## Testing, preflight, release

- **Unit tests** use the Swift Testing framework (`import Testing`, `@Suite`/`@Test`). State-detection tests run against real captured `tmux capture-pane` frames in `Tests/CCornEngineTests/Fixtures/panes/`, read at runtime via `#filePath` (never bundled). Run with the `xcodebuild test` command above.
- **Preflight** (`scripts/preflight/`) is pre-release verification agents can run end to end. `run.sh` is the CLI pane-contract test: it asks whether the installed Claude Code CLI still renders the pane text state detection is built on, by compiling the production `StateDetector.swift` into a classifier (so the harness and the app cannot disagree by construction). The e2e scripts (`e2e-auth.sh`, `e2e-chaos.sh`, `e2e-node.sh`) drive a hermetic debug app instance with its own tmux server, support dir, and debug channel, sharing nothing with a normally-running CCorn. See `scripts/preflight/README.md`. A hard failure means the CLI changed a render CCorn depends on: fix the detector and promote the captured frame into the pane fixtures so the unit suite pins it too.
- **Release** ships a signed, notarized, stapled `CCorn.app`. Bump `MARKETING_VERSION` in `project.yml` (the script reads the version from there), run preflight, then `scripts/release.sh` (needs `CCORN_SIGN_IDENTITY` and `CCORN_NOTARY_PROFILE`; credentials live in the keychain and environment, never the repo). The script finishes with `release-gatecheck.sh`, which verifies the artifact carries no debug surface and passes `codesign --verify --strict`, `spctl`, and `stapler validate`. Full procedure in `RELEASING.md`.
- **Visual / animation verification (what an agent can and cannot screenshot).** The in-app debug `shoot` command (`DebugStage`, NSView `cacheDisplay`) needs no permission and renders the sidebar faithfully (it's an opaque fill, not vibrancy), but captures **end-state only**: it reads model-layer values, so a mid-animation frame (e.g. the working breath's peak) can never be captured this way. External `screencapture -l <windowid>` (the `windowid` debug command + `e2e-lib.sh` `shot()`) captures live compositor pixels, but needs Screen Recording (TCC) permission on whatever process the shell runs under; **from a headless/agent shell it typically fails** ("could not create image from window/display", which is the TCC wall, not a bad window id — a plain full-screen `screencapture` fails the same way). So, to verify: a *static* render → `shoot`; that an *animation actually runs* → its CPU cost (a `repeatForever` loop burns measurable CPU while the window is on-screen and ~0% once the row-motion gate goes off, since the gate stops the loop when the window leaves the screen); to *watch* motion → stage a hermetic instance (the `e2e-lib.sh` pattern) and screen-record it interactively. Don't burn time trying to make `screencapture -l` work non-interactively.
